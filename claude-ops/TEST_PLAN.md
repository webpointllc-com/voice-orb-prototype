# TEST_PLAN — what Playwright covers and why

Tests run programmatically against the orb with **real audio piped through
Chrome's virtual mic**. No human in the loop, no manual click-through.

## Pattern (from forum research, verified working)

Source: maddevs.io 2024 writeup + Microsoft Playwright issue #27436 +
LinkedIn case study on fake microphone input. All three independently
confirm this pattern works in Chromium-based Playwright.

```typescript
// playwright.config.ts
import { defineConfig } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  use: {
    launchOptions: {
      args: [
        '--use-fake-device-for-media-stream',
        '--use-fake-ui-for-media-stream',
        // Per-test audio is set via env var read by the spec
      ],
    },
    permissions: ['microphone'],
  },
});
```

Per-test audio swap is done by launching a fresh browser context with the
file path interpolated into `--use-file-for-fake-audio-capture` — Playwright
issue #27436 noted that setting this globally then changing it per-test
breaks audio playback. Each test gets its own browser instance.

## Test fixtures

- `tests/fixtures/pass.wav` — recording of "my voice is my password"
- `tests/fixtures/fail.wav` — recording of "hello there how are you"
- `tests/fixtures/silence.wav` — 3 seconds of silence

All three are 16 kHz mono PCM WAV. Generated by
`scripts/generate-fixtures.sh` using `say` (macOS) or `espeak-ng` (Linux).
**Fixtures are NOT committed to git** (binary, regeneratable). The
fixtures README explains how to make them.

## Test cases (in `tests/voice-orb.spec.ts`)

| # | Name | Audio | Expected end state | Notes |
|---|---|---|---|---|
| 1 | renders the orb on load | n/a | DOM has `#wave` canvas + `#micPrompt` | Sanity check |
| 2 | shows the LISTENING state when mic is granted | silence.wav | `#stateText` = "Listening" within 2s | Mic auto-start path |
| 3 | the prompt phrase pulses when audio detected | pass.wav | `.prompt-phrase` has `.active` class within 1s of audio start | RMS threshold path |
| 4 | the passphrase triggers RESPONDING | pass.wav | `#stateText` = "AI responding" within 5s | THE happy path |
| 5 | a wrong phrase triggers FAILED | fail.wav | `#stateText` = "Try again" within 6s | THE fail path |
| 6 | RESPONDING returns to LISTENING after hold | pass.wav | `#stateText` returns to "Listening" within 4s | Verdict release |
| 7 | FAILED returns to LISTENING after hold | fail.wav | `#stateText` returns to "Listening" within 5s | Verdict release |
| 8 | no console errors during a full pass cycle | pass.wav | `page.on('pageerror')` never fires | Presentation hygiene |
| 9 | /api/state endpoint is hit on transitions | pass.wav | Route handler called ≥3 times (LISTENING, THINKING, RESPONDING) | Backend wiring |
| 10 | the orb is keyboard-accessible | n/a | `Tab` lands on `#micBtn`, `Space` toggles it | A11y baseline |

Test 9 requires the Express server running on `localhost:3000`. The other
tests can run against the static file or the dev server interchangeably.

## Running the suite

```bash
npm install
npx playwright install chromium       # first time only
bash scripts/generate-fixtures.sh     # makes tests/fixtures/*.wav
npm test                              # runs voice-orb.spec.ts
npm run test:ui                       # interactive runner (debugging)
npm run test:report                   # open the HTML report
```

CI: GitHub Actions config (Phase C) will run the same commands on every
PR. Coverage gate: all 10 tests must pass for merge.

## Why these tests catch the things that have burned Drew

- **Test 4 + 5** — the verdict path. Drew has had iterations where the
  orb LOOKED right but never actually transitioned to RESPONDING because
  the phrase gate was broken. These two tests are the smoke alarm.
- **Test 8** — no console errors. Corporate environments often have devs
  open the inspector mid-demo. A red error count is the kind of detail
  that lands "mom and pop" judgment.
- **Test 9** — the `/api/state` wiring. The orb already fires these events
  client-side. The test ensures the server actually receives them, which
  is the whole point of having a backend at all.
- **Test 10** — keyboard accessibility. Required for accessibility
  compliance (Drew's company explicitly serves an accessibility-first
  market). Catching regressions here protects the doctrine.

## What this plan does NOT cover (yet)

- Visual regression. Playwright supports screenshot snapshots; deferred to
  Phase C polish. The animations are time-dependent so snapshot timing
  needs deterministic seeding first.
- Cross-browser. Web Speech API is Chrome-only in practice. Firefox /
  Safari paths fall back to `SIMULATE` which doesn't have a verdict.
  Tested manually before each release.
- Load / concurrency. One user at a time on the corporate stage. Render
  hobby tier handles that without help.
- Network failure injection. Phase D once a baseline is green.
